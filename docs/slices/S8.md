## 🎯 Prompt (paste into `/design-slice`)

```
/design-slice S8 — Redeem points (member redemption with a reward reference)

Members can REDEEM points for a reward — a self-service deduction that records WHAT the points were
redeemed for. Distinct from a generic "spend": it's member-initiated and carries a reward reference.

REST cycle:
- POST /accounts/{account_id}/redeem    (member: OWN account only, else 403 · admin: any)
  body: { "ref":"rdm-1", "points":150, "reward":"R50 voucher", "occurred_at":"2024-06-01T10:00:00Z" }
    → 201 the recorded redemption (kind=redeem, with its reward)
    → 409 insufficient_balance if it would drive the balance below zero
    → 200 idempotent replay on a known ref (counted once)
    → 404 unknown account · 403 a member redeeming someone else's account

Data:
- widen transactions.kind CHECK from ('earn','spend') to ('earn','spend','redeem'). SQLite can't
  ALTER a CHECK, so this is a TIMESTAMPED goose migration that REBUILDS the table (create new → copy
  → drop → rename), preserving UNIQUE(ref), the FK, the points>0 CHECK, and idx_transactions_account.
- add a nullable `reward` TEXT column (set only on redeem rows; null for earn/spend).

Correctness (REUSE — don't reinvent):
- balance is now Σearn − Σspend − Σredeem → UPDATE the BalanceForAccount query so redeem is a
  DEDUCTION (earn = +, spend & redeem = −). Miss this and balances silently go wrong.
- reuse the S2 ATOMIC guard: insert the redeem, recompute balance in the SAME sql.Tx, rollback → 409
  if < 0. The single writer serialises racing redeems.
- idempotent on ref (UNIQUE(ref)), exactly like earn/spend.
- identity from the VERIFIED token via context (wallet.Authorize) — never the URL.

Enforce (docs/ACCEPTANCE.md):
- INV-24 a redeem deducts points and counts against balance (Σearn − Σspend − Σredeem)
- INV-25 a redeem that would go below zero is rejected (409)
- INV-26 concurrent redeems (and redeem-vs-spend races) never overdraw; final balance exact (-race)
- INV-27 redeem is member-own / admin-any (cross-account member → 403); the reward is recorded + returned
- INV-28 redeem is idempotent on ref (replay → 200, counted once)

Notes:
- S7's GET /accounts/{id}/transactions already returns all kinds, so redeem rows surface there for
  free — just widen the `kind` enum in the OpenAPI Transaction schema to include redeem.
- A single-txn redeem is NOT audited (the brief only requires auditing batch attempts) — keep it off
  the audit path, consistent with earn/spend.

Deps: S1 + S2 + S3 merged. Stream: A (money path).
Read: docs/specifications.pdf, docs/ARCHITECTURE.md, docs/REST_API_GUIDELINES.md,
docs/ACCEPTANCE.md (INV-24..28), and the S2 spend code (the guard this reuses).
```

## 🧠 Notes
- **Why a distinct `kind=redeem`** (not just `spend`): it records the redemption intent + the reward,
  and reads as a member self-service action — while `spend` stays the generic/admin deduction.
- The table-rebuild migration is the one real cost; everything else **reuses S1/S2/S3** — the atomic
  guard, idempotency, and the access rule all come along for free.
