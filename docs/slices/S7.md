## 🎯 Prompt (paste into `/design-slice`)

```
/design-slice S7 — Listings (accounts + transactions)

Admin and members need to LIST things, not just fetch one by id. Add the collection/read endpoints.

REST cycle:
- GET /accounts                    → ADMIN-ONLY list of all accounts
    → 200 [ { "account_id", "name", "role", "balance" }, ... ]
    → 403 for a member
- GET /accounts/{id}/transactions  → that account's ledger (newest-first)
    → 200 [ { "ref", "kind", "points", "occurred_at" }, ... ]
    → member: OWN account only (else 403) · admin: ANY account
    → 404 if the account doesn't exist

No pagination — return all rows (demo scale). Note in SOLUTION.md that limit/offset or cursor
paging is the production upgrade.

Enforce (docs/ACCEPTANCE.md):
- INV-18 GET /accounts is ADMIN-ONLY (a member → 403)
- INV-19 transaction list is account-scoped: member-own / admin-any (cross-account member → 403)
- INV-20 transaction list returns ONLY that account's transactions (no cross-account leak),
  stable newest-first order

Traps:
- Reuse the S3 access rule (wallet.Authorize) — identity from the VERIFIED token, never the URL.
- The balance in the account list is the same derived Σ(earn)−Σ(spend) as GET /balance — reuse it.
- These are READS — no new migration, just new sqlc queries + handlers + spec paths.

Deps: S1 + S3 merged. Stream: A (core reads).
Read: docs/specifications.pdf (Task 2 "admins can view any account"), docs/REST_API_GUIDELINES.md,
docs/ARCHITECTURE.md, docs/ACCEPTANCE.md (INV-18..20).
```

## 🧠 Notes
- This is the natural reading of the brief's *"admins can view any account"* — enumeration, not just
  fetch-by-id — plus a member-facing transaction history.
- Sibling read endpoint: **`GET /audit`** (admin-only) lives in **S4** (it owns the audit table), not
  here — bundling it would chain these list endpoints behind S4.
