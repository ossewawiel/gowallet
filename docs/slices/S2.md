## 🎯 Prompt (paste into `/design-slice`)

```
/design-slice S2 — Spend + no-negative guard

Members can spend points; a spend that would drive the balance below zero is rejected.

REST cycle:
- POST /transactions  → spend  { "ref", "account_id", "kind":"spend", "points", "occurred_at" }
    → 201 on success
    → 409 insufficient_balance if it would go negative

Enforce (docs/ACCEPTANCE.md):
- INV-3 a spend that would make balance negative is rejected (409)
- INV-4 concurrent spends on one account never overdraw; final balance is exact ← bake -race test in NOW

Traps:
- The balance check AND the write happen in ONE SQL transaction — no read-then-write gap a second
  request can slip through.
- Reuse S1's idempotency: UNIQUE(ref) applies to spends too.
- Points are integers.

Deps: S1 merged. Stream: A.
Read: docs/specifications.pdf, docs/ARCHITECTURE.md, docs/REST_API_GUIDELINES.md, docs/ACCEPTANCE.md (INV-3,4).
```
