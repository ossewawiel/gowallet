## 🎯 Prompt (paste into `/design-slice`)

```
/design-slice S3 — Auth (JWT, member/admin)

Add JWT auth and role-based access. This runs in PARALLEL after S0; the enforcement integrates
across the wallet endpoints midstream, once they exist.

REST cycle:
- POST /token  → issue a signed JWT for an account_id + role

Middleware:
- verify Bearer JWT, HS256, method-PINNED: golang-jwt WithValidMethods(["HS256"])
- drop account_id (sub) + role into the request context

Enforce (docs/ACCEPTANCE.md):
- INV-7 a member can only read/transact on their OWN account (else 403)
- INV-8 an admin can view ANY account and apply adjustments

Traps:
- Identity comes from the TOKEN via context — NEVER from the URL or body.
- Part of this slice is switching S1/S2 handlers to read account_id from CONTEXT (they accepted it
  in the body before auth existed).
- Pin the algorithm to kill the alg:none and RS↔HS confusion attacks.

Deps: S0 merged (to mount middleware + /token). Integrates with S1, S2. Stream: B.
Read: docs/specifications.pdf, docs/REST_API_GUIDELINES.md (auth & access), docs/ARCHITECTURE.md (auth note), docs/ACCEPTANCE.md (INV-7,8).
```

## 🧠 Notes
- Why parallel-from-S0: JWT issue/verify/middleware is pure crypto + HTTP — no DB dependency. It only
  needs the server skeleton to mount onto.
- The body→context identity swap is the integration seam with the spine. Keep it small by designing
  S1/S2 handlers context-ready.
