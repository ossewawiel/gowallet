## 🎯 Prompt (paste into `/design-slice`)

```
/design-slice S1 — Accounts + Earn + Balance

The foundation transaction slice: members exist, can earn points, and their balance reflects it.

REST cycle:
- POST /accounts            → 201 create  { "account_id": "member-123", "name": "Rina" }
- GET  /accounts/{id}       → 200 read
- POST /transactions        → 201 earn    { "ref", "account_id", "kind":"earn", "points", "occurred_at" }
- GET  /accounts/{id}/balance → 200 { "account_id", "balance" }

Enforce (docs/ACCEPTANCE.md):
- INV-1 same ref counted once — UNIQUE(ref); idempotent replay returns 200 with the existing txn
- INV-5 balance = sum(earn) − sum(spend), durable across a restart
- INV-2 same ref submitted CONCURRENTLY still counts once   ← bake the -race test in NOW
- INV-6 no wire-crossing: N members hit their OWN accounts, never see each other's data ← -race test NOW

Traps:
- Balance check + write live in ONE SQL transaction. Points are integers (never floats).
- Concurrency is NOT a later slice — its tests ship HERE.
- Identity: accept account_id in the body for now, but write handlers so they can read it from the
  request CONTEXT once S3 (auth) lands — that integration should be a swap, not a rewrite.

Deps: S0 merged. Stream: A. Unlocks S2 (spend).
Read: docs/specifications.pdf, docs/ARCHITECTURE.md, docs/REST_API_GUIDELINES.md, docs/ACCEPTANCE.md (INV-1,2,5,6).
```
