## 🎯 Prompt (paste into `/design-slice`)

```
/design-slice S6 — Login (credential-based token issuance)

Give members and admins a real way to LOG IN for a JWT, instead of the S3 demo mint that hands out
tokens with no credential check. This is authentication (who you are) feeding the existing
authorization (what you may do).

REST cycle:
- POST /login  { "account_id": "member-123", "secret": "..." }
    → 200 { "token": "<HS256 JWT, role claim taken from the STORED account>" }
    → 401 invalid_credentials  (identical response for a wrong secret AND an unknown account)

Data:
- add `password_hash` (bcrypt) + `role` (default 'member') to the accounts table — goose migration,
  TIMESTAMPED
- seed these two demo accounts (bcrypt-hash the secrets in the migration / seeder):
    member:  account_id "member-123", secret "demo-member-pw"
    admin:   account_id "admin-001",  secret "demo-admin-pw"
- POST /accounts may accept an optional `secret` so new members can log in; `role` defaults to
  member (admin is seed-only — never self-assignable via the API)

Docs (REQUIRED):
- publish the seeded credentials above in README.md under a "🔑 Test credentials" table, for local
  testing / grading
- add a clear note that this is a TEMPORARY measure for the demo — full authentication (real user
  registration, no published/shared secrets) will be implemented if more time allows

Reuse from S3:
- the existing token minting (claims, HS256, TTL) — /login just gates it behind a credential check
- decide the fate of the S3 demo /token: remove it, or keep it behind a dev-only flag (recommend
  removing or gating it so nobody can mint an admin token without a credential)

Enforce (docs/ACCEPTANCE.md):
- INV-14 valid credential → 200 with a JWT whose role claim = the account's STORED role
- INV-15 wrong/unknown credential → 401, no token, secret never logged, no user enumeration
- INV-16 the issued role comes from the STORED account, never the request (no self-minted admin)
- INV-17 the secret is stored only as a bcrypt hash and is never returned in any response

Traps:
- bcrypt is golang.org/x/crypto/bcrypt — PURE GO, no CGO. Use a sane cost (10–12).
- Return identical 401s for wrong-secret vs unknown-account so you don't leak which accounts exist.
- Stay in the 3-package layout: login handler in httpapi; the credential check is a wallet service
  persisted by sqlitestore.

Deps: S1 + S3 merged. Stream: B.
Read: docs/specifications.pdf (Task 2), docs/REST_API_GUIDELINES.md (auth), docs/ARCHITECTURE.md,
docs/ACCEPTANCE.md (INV-14..17).
```

## 🧠 Notes
- The brief leaves token issuance open (*"static tokens, signed tokens, or another approach"*) — this
  is the optional realism upgrade: a **thin, honest login**, not a full user-management system (no
  signup, no password reset).
- Demonstrates the **authentication vs authorization** split: `/login` proves *who you are*; S3's
  middleware + `wallet.Authorize` decide *what you may do*.
